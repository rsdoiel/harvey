Towards a Scholarly Memory Design for Harvey

Status: Draft
Date: 2026-06-10

🎯 Overview

Harvey is a secure, local language model agent designed for library and archival settings, working directly with digital objects (PDFs, images, video) and their metadata. This document proposes a scholarly-native memory architecture that preserves academic integrity, provenance, and contextual understanding across three core memory systems.

The design is informed by recent scholarship on AI in academic search:

  • Tay, Aaron. (2026). What a year of testing & thinking about AI academic search taught me. Aaron Tay’s Musings about Librarianship. https://doi.org/10.59350/rcm1j-5nx94
  • Tay, Aaron. (2026). How should academic retrieval augmented generation (RAG) systems handle retracted works? Aaron Tay’s Musings about Librarianship. https://aarontay.substack.com/p/how-should-academic-retrieval-augmented
  • Toler, Todd. (2025). Preserving Scholarly Signals in RAG Systems. Emphasizes provenance and chain of custody in retrieval-augmented generation systems.
  • PurePub.AI. (2025). Infrastructure as the Foundation of Trustworthy AI. White paper on building reliable AI systems on solid infrastructure.

📖 1. CONVERSATION MEMORY

Purpose: Maintains dialogue history, session context, and user preferences to enable coherent, context-aware interactions.

1.1 Design Principles

Principle Implementation Scholarly Rationale
Session Isolation Separate contexts per user/session Prevents cross-contamination of research threads
Temporal Decay Gradual forgetting of old context Matches human working memory limitations
Structured Context JSON-based context windows Enables precise reference to prior interactions
Citation Tracking Maintains chain of cited sources Critical for scholarly attribution

1.3 Provenance Integration

This starts with retrieval of information and looking for signals of provenance. This might be C2PA in the case of digital object retrieval but also might include inference based on the URL the data was retrieved from (in the case of retrieving content from the web). In the case of PDFs this might include a step that looks for signals like a DOI, ROR, or other unique identifiers embedded either as metadata in the PDF or in the content of the PDF itself.

In the case of memories, the key is to create provenance and maintain citation ability as the memories are formed and stored.

The area of conversational memory will need further exploration. While a turn-based citation model seems overly burdensome, a session might be a unit worth citing.

C2PA Connection: Each session could be cryptographically signed to ensure immutability along with inherent metadata already captured in the session Fountain document format Harvey uses. Similarly, enhancing the metadata around the conversational memory, RAG, and knowledge base could allow us to create a more complete citation context and provenance chain. This would be particularly helpful if Harvey had a mechanism to export the three memory types into a new model system.

1.4 Knowledge of Trusted Sources

In library and archival settings we regularly use trusted sources of metadata. This includes those tied to specific identifiers like DOI, ORCID, ROR, RAiD, ISSN, ISNI, CrossRef, DataCite, and others. Since one of the motivations of Harvey is to provide a path forward using language model systems in the context of the academic setting without the baggage of commercial language model systems, it makes sense to leverage the predictable metadata elements when they occur. This even benefits the developer setting since tools leverage metadata for discoverability such as GitHub’s CITATION.cff and software citation via codemeta.json files.

Example code use case: Harvey’s repository contains both codemeta.json and CITATION.cff in them. That is true of all the projects I’ve assigned Harvey to work on. These files can provide an element of context whenever Harvey interacts with a new Git repository in its workspace. The metadata objects like codemeta.json files include URLs as well as ORCID that can provide additional context for evaluating provenance of the content.

Finally, relying on scholarly identifiers opens the door for leveraging those systems like DataCite, CrossRef, ROR, RAiD, and others to enhance the evaluation process not just in the language model we’re running but programmatically too. For example, you can use a DOI to determine if a journal paper has been retracted by checking retraction databases.

1.5 Scholarly Use Cases

There are two use cases I think are important for Harvey outside of a coding assistant. First is as a smart search tool for exploring scholarly topics. The second is to treat a specific paper or other scholarly object as something that can be interrogated when exploring a subject. Followed up with curating metadata around the exploration would help distinguish Harvey from commercial offerings.

  • Research session continuity: Maintain context across multi-turn literature review sessions
  • Citation management: Track all sources referenced in a conversation
  • Methodology documentation: Record the search strategy and evolution
  • Collaboration support: Shareable conversation states with provenance

📚 2. RAG MEMORY (Retrieval-Augmented Generation)

Purpose: Stores and retrieves structured chunks of scholarly content with full provenance and contextual metadata.

2.1 The Scholarly RAG Problem

As identified by Todd Toler, traditional RAG systems lose scholarly signals when decomposing documents:

  • ❌ Citations and reference networks are broken
  • ❌ Version information (preprint vs. published) is lost
  • ❌ Peer review status is disconnected
  • ❌ Disciplinary conventions are ignored
  • ❌ Retractions and corrections are invisible

2.2 Harvey’s Scholarly-Aware RAG Design

2.2.1 Structural Chunking

Instead of arbitrary text windows, Harvey chunks by document structure:

Document: "AI in Academic Search.pdf"
├── Chunk 001 [TYPE: abstract] 
│   ├── Text: "This paper examines..."
│   ├── Metadata: {"section": "abstract", "page": 1, "doi": "..."}
│   ├── Citations: []
│   └── C2PA: {"manifest_hash": "...", "verified": true}
│
├── Chunk 002 [TYPE: introduction]
│   ├── Text: "Academic search has evolved..."
│   ├── Metadata: {"section": "introduction", "page": 2-3}
│   ├── Citations: ["10.1234/abc", "10.5678/def"]
│   └── C2PA: {"manifest_hash": "...", "verified": true}
│
├── Chunk 003 [TYPE: methods]
│   ├── Text: "We tested 15 search systems..."
│   ├── Metadata: {"section": "methods", "page": 4-5}
│   ├── Citations: ["10.9012/ghi"]
│   └── C2PA: {"manifest_hash": "...", "verified": true}
│
└── Chunk N [TYPE: references]
    ├── Text: "[1] Smith, J. (2024)..."
    ├── Metadata: {"section": "references"}
    ├── Citations: []
    └── C2PA: {"manifest_hash": "...", "verified": true}

2.2.3 Citation Network

Harvey builds a bidirectional citation graph within RAG Memory:

Document A (DOI: 10.xxxx/1) 
  ├── Cites: Document B (DOI: 10.xxxx/2)
  │     └── Context: "supporting argument"
  │
  ├── Cites: Document C (DOI: 10.xxxx/3)
  │     └── Context: "contradictory evidence"
  │
  └── Cited By: Document D (DOI: 10.xxxx/4)
        └── Context: "building upon"

This enables citation-aware retrieval: “Find papers that cite this method” or “Show me documents that contradict this finding.”

2.3 PDF Processing Pipeline

Harvey’s RAG Memory is populated through a scholarly-aware PDF processing pipeline:

Input: paper.pdf (with optional embedded C2PA)
       ↓
┌─────────────────────────────────────┐
│  1. INGEST & VERIFICATION             │
│  - Check for existing C2PA manifest   │
│  - Verify cryptographic signature     │
│  - Validate document integrity        │
└─────────────────┬───────────────────┘
                  ↓
┌─────────────────────────────────────┐
│  2. METADATA EXTRACTION               │
│  - DOI, title, authors (pdf_scholarly_metadata) │
│  - Version detection (pdf_version_detector) │
│  - Citation extraction (pdf_citation_extractor) │
│  - Structural parsing (pdf_structure_parser) │
└─────────────────┬───────────────────┘
                  ↓
┌─────────────────────────────────────┐
│  3. STRUCTURAL CHUNKING               │
│  - Split by sections (not arbitrary text) │
│  - Preserve section hierarchy         │
│  - Attach metadata to each chunk      │
└─────────────────┬───────────────────┘
                  ↓
┌─────────────────────────────────────┐
│  4. SEMANTIC INDEXING                 │
│  - Generate embeddings                │

                  ↓

🧠 3. KNOWLEDGE BASE

Purpose: Stores factual information, concepts, rules, and domain knowledge extracted from documents and external sources.

3.1 Scholarly Knowledge Representation

3.4 Reasoning with Knowledge Base

Harvey uses the Knowledge Base for:

  1. Concept Disambiguation:
    • “What is ‘Deep Research’ in academic search?” → Returns definition with sources
  2. Relationship Queries:
    • “How does C2PA relate to scholarly integrity?” → Traverses knowledge graph
  3. Evidence-Based Answers:
    • “What evidence supports the claim that agentic search is better?” → Returns sourced claims with C2PA verification
  4. Contextual Understanding:
    • “Explain AI academic search to a librarian” → Adapts explanation using disciplinary context from Knowledge Base

🔐 4. PROVENANCE MEMORY (C2PA-Based)

Purpose: Provides tamper-proof, cryptographic verification of all digital objects, processing steps, and interactions within Harvey.

4.1 Why Provenance Memory?

Traditional memory systems cannot provide: - ❌ Cryptographic proof that data hasn’t been tampered with - ❌ Tamper-evident audit trails for all processing steps - ❌ Verifiable chain of custody from original document to final output - ❌ Cross-system verification with other C2PA-compliant tools

C2PA solves these problems by providing a standardized way to embed and verify provenance information.

4.4 Integration with Other Memory Types

4.4.1 RAG Memory Integration

// RAGChunk represents a document chunk stored in RAG Memory
type RAGChunk struct {
    ChunkID    string                 `json:"chunk_id"`
    Text       string                 `json:"text"`
    Embedding  []float64              `json:"embedding"`
    Metadata   map[string]interface{} `json:"metadata"`
    C2PA       C2PAReference          `json:"c2pa"`
}

type C2PAReference struct {
    ManifestID          string `json:"manifest_id"`
    Verified            bool   `json:"verified"`
    VerificationTimestamp string `json:"verification_timestamp"`
}

// When storing a chunk in RAG Memory:
chunk := RAGChunk{
    ChunkID:    "...",
    Text:       "...",
    Embedding:  []float64{...},
    Metadata:   map[string]interface{}{...},
    C2PA: C2PAReference{
        ManifestID:          "manifest_001",
        Verified:            true,
        VerificationTimestamp: "2026-06-10T14:30:00Z",
    },
}

// Verification at retrieval time:
func retrieveChunks(query Query) []RAGChunk {
    chunks := semanticSearch(query)
    
    // Filter by provenance status if requested
    if query.ProvenanceRequired {
        var verifiedChunks []RAGChunk
        for _, c := range chunks {
            if verifyC2PA(c.C2PA.ManifestID) {
                verifiedChunks = append(verifiedChunks, c)
            }
        }
        return verifiedChunks
    }
    
    return chunks
}

4.4.2 Conversation Memory Integration

// ConversationEntry represents a single turn in a conversation
type ConversationEntry struct {
    Turn         int      `json:"turn"`

    HarveyOutput string   `json:"harvey_output"`
    Citations    []string `json:"citations"`
    // Provenance Memory reference
    C2PAManifestID string `json:"c2pa_manifest_id"`
    InputHash      string `json:"input_hash"`
    OutputHash     string `json:"output_hash"`
}

// When storing a conversation turn:
entry := ConversationEntry{
    Turn:          5,
    UserInput:     "...",
    HarveyOutput:  "...",
    Citations:     []string{...},
    C2PAManifestID: "conversation_manifest_005",
    InputHash:     "sha256:...",
    OutputHash:    "sha256:...",
}

// Create conversation manifest:
func createConversationManifest(sessionID string, turn int, inputHash, outputHash string) string {
    manifest := C2PAManifest{
        Type: "c2po:Manifest",
        Claims: []C2PAClaim{
            {
                Type:      "c2po:UserInputAction",
                Action:    "user_input",
                Target:    map[string]string{"hash": inputHash},
                Timestamp: time.Now().UTC().Format(time.RFC3339),
            },
            {
                Type:      "c2po:GenerationAction",
                Action:    "generate_response",
                Target:    map[string]string{"hash": inputHash},
                Result:    map[string]string{"hash": outputHash},
                Timestamp: time.Now().UTC().Format(time.RFC3339),
            },
        },
    }
    manifestID := storeInProvenanceMemory(manifest)
    return manifestID
}

🚨 10. SECURITY AND PRIVACY CONSIDERATIONS

10.1 Security Model

  • Local Execution: All processing happens on Harvey’s host machine
  • No Cloud Dependencies: No data leaves the local environment (except optional external API calls)
  • Cryptographic Verification: All objects and processing steps are verifiable
  • Key Management: Harvey’s signing keys are stored securely (HSM or encrypted local storage)

10.2 Privacy Guarantees

  • No Data Leakage: Conversations and documents stay local
  • No Telemetry: Harvey does not phone home with usage data
  • User Control: Users control what is ingested and processed
  • Auditability: All operations are logged and verifiable

10.3 Access Control

  • User Authentication: Integrate with institutional SSO
  • Document-Level Permissions: Control who can access which documents
  • Manifest Signing: Ensure only authorized users can create manifests
  • Verification Rights: Control who can verify vs. who can modify

  1. Preserves scholarly integrity through cryptographic provenance (C2PA)

  3. Enables deep understanding through a knowledge graph of concepts and relationships
