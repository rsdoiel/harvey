# Memory Mining for Skill Suggestions — Design

**Status (2026-06-30):** Design settled. See
[skill-suggest-plan.md](skill-suggest-plan.md) for the implementation plan.

---

## Background

`/memory mine` analyses session transcripts and extracts typed experience
records: pitfalls, workarounds, recommendations, patterns, user preferences.
These go into `agents/memories/` as Fountain files and are recalled
automatically at session start.

Skills are a different artefact: reusable, named, multi-step workflows stored
as `agents/skills/<name>/SKILL.md`. They are invoked explicitly by the user
via `/skill <name>` and can carry variables, compatibility metadata, and
step-by-step instructions.

**The gap:** sessions often contain multi-step workflows that are worth
repeating but are too structured for a single memory record. There is currently
no mechanism to extract these from session recordings and turn them into skills.

---

## Goals

1. `/skill suggest [session-file]` reads a session transcript and proposes
   multi-step, reusable workflows as skill candidates.
2. For each accepted candidate, Harvey writes a complete `SKILL.md` stub to
   `agents/skills/<name>/SKILL.md` — immediately usable.
3. The existing `/memory mine` flow is unchanged.
4. Skills are written to the live `agents/skills/` directory; no staging area.

---

## Non-Goals

- Auto-invoking skill suggestion without user request.
- Modifying `memory_miner.go` or the manifest system.
- Generating `scripts/compiled.bash` / `scripts/compiled.ps1` automatically
  (that is handled by the existing `compile-skill` skill).

---

## Design

### Command surface

```
/skill suggest [session-file]
```

With no `session-file`, Harvey uses the most recent `.spmd` file in
`agents/sessions/`. With a path, it reads that specific file.

Output: one proposed skill per candidate, followed by a Y/n prompt:

```
Proposed skill: setup-experiment
  Setup a new Laboratory experiment with git, codemeta.json, and
  agents/ directory structure.
  Variables: EXPERIMENT_NAME

Accept? [y/n/q] y
  Written: agents/skills/setup-experiment/SKILL.md
```

### LLM extraction prompt

A new system prompt (`skillSuggestorPrompt`) asks the LLM to identify
recurring, multi-step workflows in the session transcript and return a JSON
array. Each element is a skill candidate:

```json
{
  "name": "setup-experiment",
  "description": "One-sentence description for the SKILL.md name field",
  "long_description": "2-3 sentences explaining what the skill does",
  "variables": [
    {
      "name": "EXPERIMENT_NAME",
      "type": "string",
      "description": "Name of the new experiment",
      "example": "harvey"
    }
  ],
  "steps": [
    "Create the experiment directory",
    "Run git init",
    "Copy codemeta.json from template"
  ]
}
```

Rules for the LLM:
- Only propose workflows with ≥ 3 distinct steps
- Only propose things that would be reused across sessions (not one-off tasks)
- Prefer to propose 1-3 candidates; 0 is acceptable if nothing qualifies
- Output `[]` if the session contains no reusable workflow candidates

### SKILL.md generation

When the user accepts a candidate, Harvey writes
`agents/skills/<name>/SKILL.md` using a template:

```markdown
---
name: <name>
description: |
  <long_description>
license: AGPL-3.0
compatibility: harvey
metadata:
  author: <git-config-user-name or "unknown">
  version: 0.1.0
variables:
  <VARIABLE_NAME>:
    type: string
    description: <variable.description>
    example: "<variable.example>"
---

<long_description>

## Steps

1. <step 1>
2. <step 2>
...

---

*Generated by `/skill suggest`. Review and refine before use.*
```

### Session file selection

| Condition | Behaviour |
|---|---|
| Path given | Read that file directly |
| No path, sessions dir has files | Use the most recently modified `.spmd` |
| No path, sessions dir empty | Print "no sessions found" and exit |
| Workspace is nil | Print "no workspace open" and exit |

### Implementation location

- System prompt constant (`skillSuggestorPrompt`) → new file `skill_suggestor.go`
- `Suggestor` struct with `Suggest(ctx, sessionPath, agent, out, in) error` → same file
- `SKILL.md` template function `renderSkillMD(candidate) string` → same file
- `/skill suggest` subcommand dispatch → `commands_skill.go` in `cmdSkill`

### Subcommand registration

The existing `/skill` command already has `Subcommands`. Add `"suggest"`:

```go
"suggest": "/skill suggest [session-file] — propose skills from a session recording",
```

---

## Acceptance criteria

- `/skill suggest` with no sessions prints a friendly error.
- `/skill suggest` uses the most recent session when no file given.
- `/skill suggest agents/sessions/foo.spmd` reads that session.
- For each accepted candidate, `agents/skills/<name>/SKILL.md` is created.
- `agents/skills/<name>/` directory is created if it doesn't exist.
- `go test ./...` passes.
